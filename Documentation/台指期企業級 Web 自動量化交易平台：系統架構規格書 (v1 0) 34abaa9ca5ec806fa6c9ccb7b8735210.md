# 台指期企業級 Web 自動量化交易平台：系統架構規格書 (v1.0)

## 1. 專案願景與轉型目標 (Project Vision & Transformation)

本規格書旨在將現有的「台指期 Python 量化交易 MVP 系統」全面升級為雲端原生（Cloud-native）企業級 Web 交易平台。此次轉型不僅是技術堆棧的更換，更是交易哲學的昇華：從單機、高耦合的腳本驗證工具，演進為具備高可用、低延遲與嚴格治理能力的交易作業系統（Trading OS）。

轉型的核心支柱在於實施「控制平面 (Control Plane)」與「交易資料平面 (Trading/Data Plane)」的物理與邏輯分離。控制平面負責 UI、策略管理與審計，而交易資料平面則專注於行情處理、風險檢查與訂單執行。這種設計能有效達成 **延遲隔離 (Latency Isolation)**，確保管理端的流量壓力不會干擾毫秒級的交易執行路徑。

### 1.1 MVP 版與企業 Web 平台版關鍵指標對比

| 技術指標 | MVP 版 (單機架構) | 企業 Web 平台版 (雲端原生架構) |
| --- | --- | --- |
| **儲存架構** | 本地 SQLite 檔案 | 分層資料湖 (S3) + 時序庫 (TimescaleDB) + OLAP (ClickHouse) |
| **調度機制** | 本地 Cron / Task Scheduler | K8s CronJob + Airflow 任務編排 + 數據品質閘門 |
| **持倉管理** | 本地狀態檔 (Local State) | 事件溯源 (Event Sourcing) + 帳戶即時對帳服務 |
| **環境隔離** | 簡單參數切換 | 物理隔離的 Research, Backtest, Paper, Shadow, Live 環境 |
| **交易執行** | 策略與券商 SDK 強耦合 | 訊號與執行解耦，經由 OMS/EMS 與多層風控檢查 |
| **穩定性設計** | 單點失效 (SPOF) | K8s 多副本自癒、Broker Adapter 熔斷機制 (Circuit Breaker) |

## 2. 雲端原生總體架構設計 (Overall Cloud-Native Architecture)

本平台採用 Kubernetes (K8s) 作為基礎設施底座，利用其彈性調度能力管理複雜的量化任務。

### 2.1 基礎設施與微服務編排

系統根據運算特性將容器化服務部署於不同的 K8s 節點組：

- **策略 Runner (Strategy Runners)：** 運行於高優先級、資源預留節點，確保訊號產生不受干擾。
- **回測 Worker (Backtest Workers)：** 利用 K8s HPA (Horizontal Pod Autoscaler) 進行彈性擴縮，支持大規模參數掃描 (Parameter Sweep)。
- **交易網關群：** 包含 OMS、風控引擎與 Broker Adapter，採用低延遲網絡插件優化執行路徑。

### 2.2 事件驅動與訊號流 (Signal Flow)

為達成高度解耦，系統以 Kafka 作為通訊骨幹。標準訊號流向如下：

1. **Strategy Runner** 產生目標曝險訊號並寫入 Kafka `Signals` Topic。
2. **Risk Engine** 訂閱訊號進行 Pre-trade 檢查。
3. **OMS** 接收通過風控的訊號，根據狀態機生成訂單。
4. **Broker Adapter** 透過券商 API 送出指令。
5. **Execution Report** 逆向經由 Kafka 回傳至 **Portfolio Service** 進行部位更新。

### 2.3 前端監控塔台

採用 Next.js 與 React 構建，透過 WebSocket 實時接收 Kafka 流轉之行情與委託狀態。前端界面嚴禁直接與券商 SDK 交互，所有指令必須通過 API Gateway 的身份驗證（OIDC/MFA）與權限校驗。

## 3. 分層資料平台與連續月處理 (Data Platform & Multi-Layer Storage)

金融交易數據具備高頻、不可變與時間序列特性。我們採用 Lambda 架構分層儲存以平衡成本與查詢性能。

### 3.1 資料分層定義與技術選型

| 分層 (Layer) | 儲存技術 | 職責定義 |
| --- | --- | --- |
| **Bronze (Raw)** | MinIO / S3 | 儲存原始 Tick、期交所 CSV、券商原始 JSON 回報（不可變紀錄）。 |
| **Silver (Clean)** | TimescaleDB | **實時執行狀態**：儲存清洗後的 K 線、特徵工程數據與當前交易 Session 狀態。 |
| **Gold (Feature)** | ClickHouse | **OLAP/研究**：儲存價差調整後的連續月資料，供大規模回測與因子分析使用。 |

### 3.2 連續月合約處理 (Rollover Logic)

系統必須維護「換月事件表 (Rollover Event Table)」，包含以下欄位：

- `Date` (換月日期)
- `Old_Contract` / `New_Contract` (合約代碼)
- `Price_Spread` (換月價差)
- `Adjustment_Factor` (調整系數)

回測引擎使用 ClickHouse 中的 **Back-adjusted** 資料以消除跳空缺口；實盤執行層則透過 **Session Engine** 確保在最後交易日精準映射至次月真實合約。

### 3.3 數據品質閘門 (Data Quality Gates)

所有進入 Silver 層的資料需通過校驗：

- **Session 對齊：** 處理台指期日盤 (08:45-13:45) 與夜盤 (15:00-05:00) 的時戳歸屬，解決 T+1 跨日歸帳問題。
- **異常過濾：** 偵測零價、負價或超出漲跌幅限制的異常 Tick。
- **完整性檢查：** 自動填補 K 線缺漏，確保回測不因數據缺失產生錯誤訊號。

## 4. 交易執行核心：OMS、風險引擎與代理網關 (The Trading Core)

### 4.1 OMS 訂單狀態機與冪等性

OMS 負責管理委託的生命週期，並使用 **Idempotency Key (冪等鍵)** 確保網絡波動時不會重複下單。

- **狀態轉移：** `Pending` -> `Sent` -> `Executed` / `Rejected` / `Cancelled`。
- **對帳機制：** 基於 **Event Sourcing (事件溯源)**，系統會重播 Kafka 上的成交事件，定期與券商 API 帳務進行比對 (Reconciliation)，確保部位 100% 同步。

### 4.2 三層風險控制規則

1. **Pre-trade (下單前)：** 檢查單筆最大口數、**Stale Quote Protection** (行情延遲 > 3 秒禁止下單)、**Price Band Check** (價格偏離最新成交價 > N%)。
2. **In-trade (盤中)：** 監控當日總損益 (Daily PnL)、保證金使用率、以及 **Max Order Rate** (限制每秒下單頻率)。
3. **Post-trade (日終)：** 自動產出當日交易偏離報告。

### 4.3 Broker Adapter 與熔斷設計

以 Shioaji 為首個接入口，封裝於 Adapter 層。內部實作 **Circuit Breaker (熔斷模式)**：當券商 API 連續逾時或拒絕率達門檻時，自動切換至「只收不發」降級模式，並觸發全系統告警。

## 5. 台指期多產品曝險正規化與智能調度 (TX/MTX/TMF Allocation)

系統將 TX (大台)、MTX (小台)、TMF (微台) 視為同一資產類別的拆分，基準比例為 **1 : 4 : 20**。

### 5.1 智能配置數學邏輯 (Integer Optimization)

系統 Allocator 模組將「目標目標曝險 (Target Net Delta)」轉化為具體合約組合，計算邏輯如下：

- **輸入指標：** Target Net Delta, 當前 ATR (波動率), 帳戶保證金餘額, 安全系數 (Safety Factor)。
- **優化目標：** 最小化 **殘差曝險 (Residual Exposure)** 與 **滑價成本 (Slippage Impact)**。
- **換算公式：** 每點價值 TX(200) / MTX(50) / TMF(10)。系統會根據流動性優先填滿 TX，餘數再由 MTX 與 TMF 補足。

### 5.2 交易時段與 Session 切換

系統 Session Engine 嚴格定義台指期兩大時段：

- **Regular Session:** 08:45 - 13:45
- **After-hours Session:** 15:00 - 05:00 (T+1 歸帳) 系統具備自動換日邏輯，在夜盤收盤後自動處理帳務結算與日盤預讀。

## 6. 策略生命週期管理與量化 DevOps (Strategy Lifecycle & Governance)

策略被視為軟體產品，必須通過五層環境驗證：

1. **Research:** 因子探索。
2. **Backtest:** 固定數據版本下的回測，產出 Sharpe, MDD, Sortino 等指標。
3. **Paper:** 接入即時行情但不送單至真實交易所。
4. **Shadow:** 實盤行情產生訊號並記錄理論成交，觀察 **Slippage Model Impact**。
5. **Live:** 正式實盤，需經由 Risk Manager 簽核。

## 7. 平台可觀測性、資安與權限控管 (Observability & Security)

### 7.1 全鏈路追蹤 (Full-link Traceability)

採用 **OpenTelemetry** 實作分散式鏈路追蹤。監控一個訊號從 Strategy Runner 發出，經過 Kafka、Risk Engine、OMS 到 Broker Adapter 接收成交回報的完整時延。

- **Metrics:** Prometheus 監控帳戶權益與系統負載。
- **Logs:** Loki 蒐集不可竄改的交易日誌。

### 7.2 資安防護與 RBAC 權限矩陣

所有憑證 (API Key) 儲存於 **HashiCorp Vault / Cloud KMS**。系統採嚴格角色訪問控制：

| 角色 (Role) | 權限範疇 |
| --- | --- |
| **Owner** | 憑證管理、資金劃轉、全平台配置。 |
| **Quant** | 策略開發、回測執行、提交上線申請。 |
| **Trader** | 策略啟停、手動平倉、手動改單。 |
| **Risk Manager** | 設定風控門檻、審核策略上線、操作 Kill Switch。 |
| **Auditor** | 僅限讀取稽核日誌與歷史成交紀錄。 |
| **Viewer** | 僅限觀察實時部位與損益。 |

## 8. 實作里程碑與技術驗收標準 (Implementation Roadmap & UAT)

本案採增量交付，確保每一階段皆具備 UAT (技術驗收) 價值。

- **Phase 0-1 (基礎建設)：** K8s 環境部署、資料分層架構實作。
- **Phase 2-3 (核心開發)：** Session Engine 實作、連續月資料清洗、回測平台。
- **Phase 4-5 (交易路徑)：** OMS 狀態機開發、三層風控引擎、Broker Adapter 接入。
- **Phase 6 (正式上線)：** 災難復原演練 (DR Drill)、全鏈路可觀測性上線。

### 技術驗收標準 (UAT)

1. **數據完整性：** 平台數據與期交所官方歷史資料一致性須達 100%。
2. **風控攔截率：** 模擬 Stale Quote 或超限下單時，攔截成功率須為 100%。
3. **狀態一致性：** 系統異常重啟後，透過 Event Sourcing 重建之部位須與券商端完全對齊。

**結語：** 本平台之核心哲學在於「策略產生訊號，平台負責執行」。透過本規格書定義的架構，我們將打造出一個具備金融級穩定性與高度擴展能力的台指期量化交易基石。
