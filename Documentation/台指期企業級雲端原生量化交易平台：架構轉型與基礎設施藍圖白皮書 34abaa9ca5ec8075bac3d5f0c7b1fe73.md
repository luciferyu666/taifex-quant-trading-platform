# 台指期企業級雲端原生量化交易平台：架構轉型與基礎設施藍圖白皮書

## 1. 架構演進：從單機交易腳本到量化交易作業系統

### 戰略背景引言

在量化交易的初級階段，開發者常依賴 Python 單機腳本與本地資料庫進行策略驗證。然而，當交易規模跨入機構級別時，傳統單機模式在面對高併發行情、多帳戶同步及合規審計時會顯得極其脆弱。從單機腳本轉型為雲端原生 Web 平台，並非僅是技術棧的搬遷，而是將量化交易從「個人驗證」提升至「企業級資產管理」的戰略躍遷。此轉型旨在建立一個具備防禦性設計、資源彈性調度且具備完整故障隔離能力的「量化交易作業系統」。

### 轉型價值矩陣

根據系統演進路徑，MVP 與企業級 Web 平台在核心架構維度上的對比如下：

| 維度 | 傳統單機 MVP 設計 | 企業級 Web 平台方案 |
| --- | --- | --- |
| **儲存機制** | SQLite 本地檔案儲存 | 分散式資料湖與 TimescaleDB + ClickHouse 分層儲存 |
| **調度管理** | 本地 Cron / Task Scheduler | Airflow 工作流與 K8s 容器化動態調度 |
| **狀態追蹤** | 本地 JSON/Pickle 狀態檔 | **事件溯源 (Event Sourcing)** 與持久化位置服務 |
| **回測架構** | 單機序列執行 | 分散式 Backtest Farm 與 Walk-forward 交叉驗證 |
| **風控能力** | 硬編碼於策略邏輯中 | 獨立風控引擎、**三級 Kill Switch** 與預編譯規則 |
| **安全審計** | 無或簡單 Text Log | 多租戶 RBAC、不可竄改的金融級審計日誌 |

### 分析核心價值：訊號與執行的深度解耦

本架構最核心的戰略價值在於實施「訊號產生」與「交易執行」的物理與邏輯解耦。在防禦性架構設計中，這種解耦確保了策略計算層的邏輯異常不會直接穿透至交易所。評估此轉型後的影響：首先，它實現了**故障隔離 (Fault Isolation)**，即便策略進程崩潰，執行層（OMS）仍能維持既有訂單狀態並觸發安全平倉；其次，它加速了策略迭代，研究員可專注於 Alpha 邏輯，而將底層券商 API（如處於 Alpha 階段的 Shioaji SDK）的穩定性風險交由標準化的 **Broker Adapter** 層進行緩衝與過濾。

### 轉場銜接

在確立了轉型的戰略必要性後，我們必須建立一個高度隔離且低耦合的雙平面架構，以支撐高頻數據與指令的流轉。

## 2. 核心架構設計：控制平面與交易/資料平面的分離

### 戰略背景引言

金融系統的設計哲學在於「合規監管」與「極致性能」的動態平衡。為解決此權衡問題，本平台採用「控制平面 (Control Plane)」與「交易/資料平面 (Trading/Data Plane)」分離的架構，旨在確保管理行為不干擾執行鏈路，且執行層的負載波動不影響管理層的可用性。

### 平面職責定義

- **控制平面 (Control Plane)：**
    - **治理中心：** 負責身份認證 (RBAC)、策略註冊、版本管理及參數 Schema 定義。
    - **風險配置：** 設定全平台、帳戶及策略級別的風控閾值，並管理審計日誌與 Web 控制台。
- **交易/資料平面 (Trading/Data Plane)：**
    - **高性能執行：** 即時行情接入、OMS/EMS 訂單生命週期管理。
    - **Broker Adapter 層：** 鑑於 Shioaji 等 SDK 仍具 Alpha 性質，此層負責抽象化券商接口，實現 Broker-Agnostic（券商中立）架構，降低 API 變動風險。
    - **即時風控：** 執行毫秒級的 Pre-trade 檢查，並實施分層數據的高頻寫入。

### 組件協作藍圖：異步通訊與隔離執行

兩平面間透過 API Gateway 傳遞管理指令，並利用 **Kafka 事件流** 進行非同步狀態同步。策略執行個體（Strategy Runner）被封裝在獨立的 Docker 容器中，透過 Kubernetes 進行隔離。這種設計不僅解決了資源爭奪問題，更確保單一策略的內存洩漏或崩潰不會引發系統性塌方。

### 轉場銜接

明確了功能的劃分後，支撐此架構運行的「血液」——數據基礎設施，必須具備處理海量時間序列數據的能力。

## 3. 分散式資料湖與分層儲存結構 (Data Lakehouse)

### 戰略背景引言

在量化交易中，數據一致性直接決定了回測的真實性。任何由於數據清洗不當導致的「換月跳空」或「行情漂移」，都將導致錯誤的資產定價與策略誤判。

### 四層數據架構方案

平台採用 Data Lakehouse 架構，將數據流進行分級處理：

1. **Bronze Raw (S3/MinIO):** 儲存原始 Tick、期交所 CSV 及券商 API 原始 Payload。
2. **Silver Clean (TimescaleDB):** 利用 TimescaleDB 的 **Hypertables** 與 **Continuous Aggregates** 技術，儲存經過 Session 對齊（日/夜盤）且完成清洗的 K 線。
3. **Gold Feature (TimescaleDB/ClickHouse):** 預計算指標層。**ClickHouse** 在此發揮其 OLAP 強項，負責大規模回測結果分解、績效分析及多因子特徵運算。
4. **Research Dataset:** 固定版本的不可變快照，供研究員重現實驗結果。

### Data Quality Gates：數據品質門禁

在 Silver 轉 Gold 層的過程中，系統設有自動化檢核邏輯，針對零價、負價、K 線缺漏及非交易時段數據進行自動攔截與告警，確保「髒數據」不進入策略計算鏈路。

### 連續月合約處理邏輯

台指期具備每月結算特性，系統嚴格遵循「回測使用調整資料，下單使用真實合約」原則。**換月事件表 (Rollover Event Table)** 必須包含以下關鍵參數：

- Rollover Date (換月日期)
- Basis/Spread (價差調整項)
- Old/New Symbol (合約代碼)
- Adjustment Factor (調整係數與資料版本)

### 轉場銜接

數據層為策略提供了土壤，而容器化技術則決定了這些策略如何在雲端環境中安全、高效地被孕育。

## 4. 雲端原生部署與容器化執行環境

### 戰略背景引言

Kubernetes (K8s) 作為量化基礎設施底座，其戰略價值在於提供極限的自我修復能力。對於企業級平台而言，K8s 確保了當某個 Runner 因異常掛載或網路抖動失效時，系統能自動在秒級內於健康節點重建狀態。

### 策略生命週期管理與量化驗收

每個策略必須通過嚴格的路徑，並符合量化門檻：

- **Draft/Research:** 邏輯驗證。
- **Backtested:** 必須通過壓力測試與 Walk-forward 檢驗。
- **Paper:** 即時行情驗證，確認 API 交互與回測邏輯一致。
- **Shadow (影子):** 在實盤行情下計算訊號但不送單，**驗收門檻：** 理論成交價與市價之滑價偏差必須連續 5 個交易日 < 0.1%。
- **Live (實盤):** 小規模實盤觀測後正式上線。

### 環境與憑證隔離原則

系統嚴格禁止環境混用。Paper 與 Live 必須在 Kubernetes Namespace、Vault 憑證路徑及風控規則上進行**物理實體隔離**，確保開發環境的誤操作無法觸及實盤資金帳戶。

### 轉場銜接

穩定的環境與高品質數據是基礎，但針對台指期特有合約規格的風險控管，則是資金安全的最後防線。

## 5. 台指期風險正規化與智能調度引擎

### 戰略背景引言

台指期 (TX/MTX/TMF) 特有的階梯式規格要求平台具備「風險正規化」能力。企業級平台必須將所有曝險轉換為「台指等值 (TX Equivalent)」，以實施統一的曝險管理。

### 曝險等值計算與交易時段定義

系統根據 1 TX = 4 MTX = 20 TMF 之比例進行正規化，規格細節如下：

| 商品名稱 | 代碼 | 每點價值 | 換算比例 | 交易時段 (台灣時間) |
| --- | --- | --- | --- | --- |
| **台指期** | TX | 200 TWD | 1.00 | 08:45–13:45 / 15:00–05:00 |
| **小台指** | MTX | 50 TWD | 0.25 | 08:45–13:45 / 15:00–05:00 |
| **微台指** | TMF | 10 TWD | 0.05 | 08:45–13:45 / 15:00–05:00 |

### 三層風控體系與 Kill Switch 觸發邏輯

系統實施 Pre-trade (單筆限制)、In-trade (實時損益) 與 Post-trade (對帳) 三層防護。**Kill Switch** 具備以下硬性觸發條件：

- **帳戶層級：** 當日損失 (Daily Loss) > 5% 時，立即停止所有加倉並啟動清算。
- **系統層級：** 券商 API 延遲 (Latency) > 2000ms 或數據不一致性 > 0.5% 時，自動切換至唯讀模式。
- **策略層級：** 觸發單策略 MDD 預警閾值時，實施強制停機。

### 智能調度算法

調度引擎根據「目標曝險 + 保證金 + ATR 波動率」自動配置最優合約組合。例如：當策略需增加 0.35 TX 曝險時，引擎將自動拆分為 1 口 MTX 與 2 口 TMF，而非粗略的四捨五入，從而實現精確的風險曝險控制。

### 轉場銜接

風險控管是底線，而系統的可觀測性則是我們在極端行情下維持冷靜的視角。

## 6. 全鏈路可觀測性與災難復原機制

### 戰略背景引言

量化交易最忌諱「黑盒效應」。當系統發生異常下單時，我們必須具備在數秒內回溯訊號觸發原因、風控判定過程及券商回報軌跡的能力。

### OpenTelemetry 技術棧與追蹤

整合 Prometheus (Metrics)、Loki (Logs) 與 Tempo (Traces)，實現「訊號產生 -> 風控校驗 -> OMS 轉單 -> 券商成交」的全鏈路追蹤。當發生異常時，開發者可透過 Trace ID 獲取該訂單在微服務間流轉的完整上下文。

### 事件溯源與狀態重建 (State Reconstruction)

系統基於 **Event Sourcing** 模式，所有部位異動均作為不可變事件寫入 Kafka。若 Strategy Runner 發生崩潰，系統將透過重播 (Replay) Kafka 事件流在秒級內重建當前持倉狀態，而非依賴緩慢的資料庫查詢。

### 災難復原與券商對帳 (Reconciliation)

**券商對帳** 被定義為系統的最後一道防線。系統每 15 分鐘自動比對「平台邏輯部位」與「券商帳務部位」。若發現不一致，將立即鎖定交易並發送 **Critical** 級別告警。

| 嚴重度 | 事件範例 | 處置方式 |
| --- | --- | --- |
| **Critical** | 部位不一致、API 斷線、日損超限 | 啟動 Kill Switch，鎖定全平台交易 |
| **High** | 行情延遲 > 500ms、委託拒絕率上升 | 降低交易頻率，發送管理員熱線告警 |
| **Medium** | 憑證即將到期、容器重啟 | 自動修復並記錄審計日誌 |

### 轉場銜接

技術的卓越最終是為了業務的穩定性服務。

## 7. 結論與未來擴展建議

### 戰略總結

本架構透過控制平面與交易平面的物理分離、基於容器化的策略執行環境、以及 TimescaleDB/ClickHouse 構成的資料湖，成功將脆弱的單機腳本轉化為具備金融級韌性的量化交易基礎設施。這確保了系統不僅能應對高波動行情，更能滿足合規監管與資產管理的嚴苛要求。

### 未來擴展路線圖

1. **多券商 Adapter 擴充：** 基於當前的抽象層，快速對接國內外多家期貨商，降低對單一 API 的依賴。
2. **AI/ML 整合優化：** 在 Gold Feature 層直接整合自動化特徵工程，支援模型在線推論。
3. **策略市集 (Marketplace)：** 建立標準化的外掛架構，支援第三方策略的受控接入與分潤。

**核心價值重申：** 量化交易的現代協作遠景應為：**「平台負責風控與執行，人類負責策略與研究」**。透過本藍圖的實施，企業將能把最專業的人才從繁瑣的基礎設施維護中釋放，專注於產生超額報酬的 Alpha 邏輯，從而在競爭激烈的台指期市場中建立長期的技術壁壘。
