# 台指期企業級 Web 自動量化交易平台：系統架構規格書

## 1. 專案願景與從 MVP 到雲端架構的演進

在量化交易的生命週期中，從單機腳本轉型為企業級平台是確保策略穩定性與資產安全性的必然路徑。本平台的核心使命是將原有的 Python 單機 MVP（僅能執行基礎下單循環）演進為具備「自營商等級」的高可用、可審計、可擴展架構。

這種演進不只是軟體工具的升級，更是從「低成本驗證」向「高可靠量化作業系統」的質變。下表總結了架構升級的核心維度，說明我們如何透過系統化的設計將技術轉化為市場競爭優勢：

| 關鍵維度 | 原 Python MVP 設計 | 企業級 Web 平台升級版 | 競爭優勢與商業價值 |
| --- | --- | --- | --- |
| **儲存機制** | 本地 SQLite 檔案 | 分層 Data Lakehouse (TimescaleDB / ClickHouse) | 支援**大規模參數掃描 (Parallel Parameter Sweeps)** 與數十億級 Tick 資料的高效 OLAP 分析。 |
| **任務調度** | 定時任務 (Cron) | 分散式調度 (Airflow / K8s CronJob) | 導入**資料品質門檻 (Data Quality Gate)**，確保回測與交易不因資料缺漏或偏差而失效。 |
| **狀態管理** | 本地檔案 (.json) | **事件溯源 (Event Sourcing)** | 實現交易狀態的完整可審計性，並支援系統重啟後的狀態自動重建。 |
| **策略驗證** | 簡單回測 | **分散式回測叢集 (Backtest Farm)** | 執行 **Walk-Forward Analysis** 與蒙地卡羅模擬，量化策略的過往擬合風險與穩健性。 |
| **安全性** | 明文憑證儲存 | HashiCorp Vault (Transit Engine) | 防止 API Key 外洩。憑證僅在下單簽章瞬間於 Broker Gateway 解密，極小化攻擊面。 |

此演進奠定了「控制平面」與「交易資料平面」分離的基礎。這種分離確保了管理端的高負載與複雜查詢不會干擾到極其敏感的交易下單路徑。

## 2. 總體系統架構：控制平面與交易資料平面的分離原則

為追求金融級的穩定性，本架構嚴格遵循「控制平面 (Control Plane)」與「交易資料平面 (Trading/Data Plane)」分離的原則。此設計旨在建立強大的容錯邊界，防止 Web 端的查詢流量對交易執行產生任何微秒級的延遲波動（Jitter）。

### 2.1 職責邊界定義

- **控制平面 (Control Plane)：** 管理非關鍵路徑任務，包括使用者 RBAC 權限、策略參數配置、歷史報表生成與稽核紀錄。即使此平面因高負載變慢，交易引擎仍能獨立運行。
- **交易資料平面 (Trading/Data Plane)：** 處理高頻即時任務，包含行情接入、策略 Runner、風控引擎 (Risk Engine) 與 OMS。此區域必須部署於高性能運算節點，具備最高優先權。

### 2.2 Kubernetes (K8s) 的底座價值

系統必須部署於 Kubernetes 環境，其價值不僅在於容器化，更在於：

- **自動化自我修復：** 當行情服務或策略 Runner 異常崩潰時，K8s 必須能自動重新調度並恢復服務。
- **水平擴展：** 根據回測任務的運算需求，動態擴張 Worker Pods。

### 2.3 環境隔離機制

系統必須實施實體或邏輯隔離，**嚴禁僅靠單一設定檔參數切換環境**。不同環境（Research, Backtest, Paper, Shadow, Live）必須擁有獨立的網路邊界 (Namespace)、獨立的資料庫執行體與專屬的券商 API 憑證，以防止實驗性代碼誤觸實盤環境。

## 3. 分層資料儲存與大數據治理架構

量化交易的核心資產在於資料。本系統建立分層儲存架構，旨在平衡「高頻寫入」與「高效分析」的互斥需求。

### 3.1 資料分層架構定義

| 層級 | 儲存技術 | 功能說明 |
| --- | --- | --- |
| **Bronze Raw** | S3 / MinIO | 儲存原始 Tick、期交所 CSV。作為不可變的資料來源（Immutable Source of Truth）。 |
| **Silver Clean** | TimescaleDB | 清洗後的時序資料。利用其 Hypertable 特性處理高頻寫入與即時 K 線聚合。 |
| **Gold Feature** | ClickHouse | 儲存計算後的技術指標與績效。優化 OLAP 分析查詢，縮短參數最佳化時間。 |

### 3.2 Contract Master 與連續月處理邏輯

系統必須配置專屬的 **Contract Master Service**，負責管理台指期 (TX/MTX/TMF) 的合約生命週期：

- **Rollover Event Table：** 記錄每次換月的價差 (Basis) 與調整係數。
- **存取路徑隔離：** 供策略研究用的 **Back-adjusted 連續資料** 與實盤執行用的 **Real-contract 真實代碼資料** 必須在存取路徑上嚴格分離，杜絕看穿未來 (Look-ahead bias) 或下錯無效合約。

## 4. 核心交易組件：OMS、風控引擎與 Broker Gateway

交易執行層的關鍵目標是確保策略訊號轉化為委託的過程具備「確定性」與「不可竄改性」。

### 4.1 訊號解耦與 OMS 狀態機

- **訊號解耦原則：** 策略 Runner 僅被允許輸出標準化訊號（Target Exposure，例如：+2 TX），禁止直接呼叫任何下單 API。
- **OMS 狀態機：** 每個訂單必須遵循嚴格的狀態流轉（Pending → New → Filled / Rejected），且必須附帶**冪等性鍵 (Idempotency Key)**，確保網路重連時不發生重複下單。

### 4.2 大小微台智能調度 (Allocator)

不同於簡單的 1:4:20 數學轉換，**Allocator 必須是一個最佳化模型**。它需根據當前帳戶的保證金使用率、市場即時流動性 (Liquidity) 與 ATR 波動率，將目標曝險分配至 TX、MTX 或 TMF。其核心目標是在最小化殘差曝險的同時，最大化帳戶資金利用率。

### 4.3 Broker Gateway 抽象層

以 Shioaji 為例，系統必須實作 **Adapter Pattern**。鑑於券商 SDK 可能存在不穩定性，Adapter 必須作為安全包裝層，負責處理斷線重連、API 速率限制（Rate Limiting）與異常訊息轉換，確保核心 OMS 不被特定券商 SDK 的生命週期綁架。

## 5. 多層次風控體系與安全管理

風控系統是平台的最後一道防線，必須具備 Pre-trade、In-trade 與 Post-trade 的三層防禦能力。

### 5.1 關鍵風控指標與防禦機制

- **核心指標：** 包含帳戶權益 (Equity) 監控、保證金使用率限制、最大日損 (Daily Loss Limit) 以及單一策略曝險上限。
- **緊急停機 (Kill Switch)：** 系統必須提供三路緊急停機路徑：
    1. **Web UI：** 供交易員人工一鍵平倉。
    2. **API：** 供風控服務偵測到異常（如券商斷線）時自動觸發。
    3. **CLI (Out-of-Band)：** 專門應對緊急情況，如 Web Ingress 故障或 K8s 控制台回應遲緩時，運維人員可跳過前端直接下達停機指令。

### 5.2 資安合規與憑證管理

系統必須對齊 **OWASP ASVS** 標準。所有券商憑證 (API Keys) 必須儲存於 **HashiCorp Vault**，並使用其 **Transit Secret Engine** 進行動態加密。策略 Runner 在下單時不會取得明文金鑰，僅由密碼模組完成簽章作業，確保金鑰在整個生命週期中皆處於受控狀態。

## 6. 策略生命週期與分散式回測環境

策略在平台中應被視為正式軟體資產，須經歷：**Draft → Research → Backtested → Paper Approved → Shadow Trading → Live**。

### 6.1 不可變綁定 (Immutable Binding)

為了確保回測結果可重現，系統必須實施「不可變綁定」機制。任何回測報告必須自動關聯該次實驗的 **Git Commit SHA**、**Data Snapshot 版本**與**參數 Schema**。無效綁定的回測報告將被禁止進入 Shadow 或 Live 審核流程。

### 6.2 Backtest Farm

利用 K8s Worker 建立分散式運算資源池，執行大規模的 Walk-forward 分析，藉此驗證策略在不同市場環境下的魯棒性。

--------------------------------------------------------------------------------

## 7. 可觀測性、營運監控與災難復原

在動態的交易環境中，透明度與復原力是營運的核心。

### 7.1 可觀測性技術堆疊

- **Traces (OpenTelemetry)：** 追蹤每一筆訊號從產生到下單、成交的完整鏈路。
- **Metrics & Logs：** 使用 Prometheus 收集系統延遲，並在 Grafana 建立即時監控看板。

### 7.2 災難復原 (DR) 與事件溯源

系統核心狀態管理必須採用 **Event Sourcing** 機制。

- **狀態重建：** 當系統發生故障重啟時，OMS 必須從 Kafka 重新讀取當日的成交回報（Execution Reports）事件流，重建內部的持倉狀態 (Internal Position State)。
- **對帳機制 (Reconciliation)：** 重建完成後，系統必須自動透過 Broker Gateway 與券商端實時持倉進行對帳，若發現部位不一致，必須立即觸發 Critical 告警並進入交易鎖定模式。

--------------------------------------------------------------------------------

**結語：** 本系統架構將台指期量化平台從單機實驗室環境升級為企業級交易基礎設施。透過控制平面分離、事件溯源機制與嚴密的風控體系，我們確保系統在獲取超額報酬的同時，具備應對市場極端風險的韌性與能力。
