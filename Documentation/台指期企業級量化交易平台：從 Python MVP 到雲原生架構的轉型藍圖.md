# 台指期企業級量化交易平台：從 Python MVP 到雲原生架構的轉型藍圖

## 1. 專案背景與平台轉型願景

在量化交易的生命週期中，單機腳本（MVP）是驗證策略邏輯的起點，但若要邁向「企業級營運」，系統必須從脆弱的工具轉型為具備高可用性、可稽核性與橫向擴展能力的「量化交易作業系統」。本藍圖旨在將原有的台指期 Python MVP，重構為支援多租戶、多策略並行處理的雲原生平台。這不僅是技術棧的遷移，更是建立強化的風控體系與符合金融稽核標準的合規流程，確保每一筆訊號皆可追溯，每一份部位皆具備最終一致性。

### 核心轉型對照

下表定義了從「單機腳本」演進至「雲端作業系統」的技術斷代：

| 維度 | 原 MVP 設計 | 企業級 Web 平台升級版 |
| --- | --- | --- |
| **資料儲存** | SQLite 儲存歷史 K 線 | Data Lake + TimescaleDB + ClickHouse 分層儲存 |
| **任務排程** | Cron 手動更新資料 | Airflow/Dagster + K8s CronJob 自動化管線 |
| **狀態管理** | 本地狀態檔（State File） | **Event Sourcing** + Position Service + Broker Reconciliation |
| **回測環境** | 單機 Backtrader / VectorBT | 分散式 Backtest Farm + 資料版本化快照 |
| **券商對接** | 直接依賴單一券商 SDK | Broker Gateway 抽象層（支援 Shioaji、富邦 Neo 等） |
| **安全控管** | 單人操作，憑證寫死 | **RBAC + ABAC**、MFA、HashiCorp Vault 憑證管理 |
| **稽核追蹤** | 基本文字 Log | **Immutable WORM 儲存**之不可竄改稽核日誌 |

### 策略重要性剖析

轉型為平台化的關鍵價值在於實現「策略與執行解耦」。系統不再僅是執行買賣的腳本，而是轉變為一個資源調度器，負責多租戶隔離與動態曝險管理。這種架構能確保不同研究員在共享基礎設施下互不干擾，並透過平台化的風控引擎建立起一道強制性的「系統停機開關 (Kill Switch)」，將量化交易從個人化的技術驗證提升為可規模化的作業資產。

## 2. 雲原生總體架構與微服務切分

為確保在台指期高負載開盤期間的系統穩定性，架構設計核心在於「控制平面 (Control Plane)」與「交易/資料平面 (Trading/Data Plane)」的物理分離。

### 雙平面架構價值

- **控制平面：** 負責使用者身分驗證、策略配置、權限審批與報表呈現。此平面即使負載突增，亦不會阻塞交易執行路徑。
- **交易/資料平面：** 專注於即時行情解析、風險引擎毫秒級判定、以及 OMS 訂單狀態維護，確保極低延遲與高並發穩定性。

### 微服務職責矩陣與安全性規範

本平台採用 Kubernetes 作為核心**雲端編排底座 (Orchestration Substrate)**，以支援宣告式部署與具備自我修復 (Self-healing) 能力的狀態化工作負載：

- **Identity Service：** 整合 **RBAC (角色存取控制)** 與 **ABAC (屬性存取控制)**，依據使用者角色、時段與帳戶狀態進行動態授權。
- **Strategy Registry：** 管理策略的版本生命週期、參數 Schema 與 Docker Image 鏡像版本。
- **Market Data Service：** 提供標準化即時行情推播與歷史 K 線聚合查詢。
- **Risk Engine：** 執行三層風控檢查，作為所有訂單送出前的唯一檢查閘道。
- **OMS (Order Management System)：** 負責訂單狀態機維護，並透過 **Event Sourcing** 模式完整記錄狀態變更序列。
- **EMS / Broker Gateway：** 進行多券商協議轉換（如將標準化訊號轉換為 Shioaji API 指令）。

## 3. 多層級資料湖與時間序列引擎設計

在量化交易中，「資料版本化」是確保回測重現性與實盤一致性的基石。若無嚴謹的資料治理，策略的績效將流於隨機。

### 資料分層架構圖

1. **Bronze Raw (S3/MinIO)：** 儲存期交所原始 CSV 與券商 API 原始回應。此層為「不可變 (Immutable)」來源，禁止修改。
2. **Silver Clean (TimescaleDB)：** 儲存經清洗與時段對齊的 K 線與 Tick 資料，利用 Hypertable 提供高效能時間序列查詢。
3. **Gold Feature (ClickHouse)：** 儲存計算後的技術指標、波動率因子與連續月合約資料。利用其列式儲存特性支撐大規模數據分析與 Backtest Farm 運算。

### 連續月合約處理邏輯

根據來源文獻，系統必須同時支援並精確區分以下兩類調整資料：

- **Back-adjusted (差值調整連續月)：** 透過加減固定差值消除換月跳空，適用於大多數趨勢策略回測。
- **Ratio-adjusted (比例調整連續月)：** 透過乘法係數調整，適用於需要波動率正規化或長期比例研究之策略。
- **真實合約價格：** 所有實盤下單指令必須回歸真實代碼（如 TX2606）與當下真實價格。 所有換月細節皆記錄於 **Rollover Event Table**，包含換月日期、新舊合約價差、調整係數與資料版本，以確保回測與實盤無縫銜接。

## 4. 交易執行、OMS 與大小微台調度策略

企業級系統的核心在於將「交易決策」與「訂單執行」徹底隔離。策略僅產出目標曝險訊號，具體執行路徑由平台統籌。

### 大小微台 (TX/MTX/TMF) 調度模型

平台根據台指期 1:4:20 的曝險比例，將原有邏輯升級為**「目標曝險整數最佳化 (Integer Optimization) 模型」**：

- **輸入參數：** 目標曝險金額 (Net Delta)、目前持倉、保證金使用率、以及 **ATR 波動率**。
- **計算邏輯：** 系統以**「最小化交易殘差 (Minimizing Residuals)」**為目標函數，自動計算最佳合約組合（例如當目標曝險為 1.15 TX 時，自動拆分為 1 TX + 1 MTX + 3 TMF）。這確保了資金效率最大化，並減少滑價損失。

### OMS 訂單狀態機與對帳機制

- **狀態管理：** OMS 使用 **Event Sourcing** 技術記錄每張訂單從 Pending、Accepted 到 Filled 的每一筆事件軌跡，確保系統崩潰後能完全復原持倉狀態。
- **券商對帳 (Broker Reconciliation)：** 平台定期與券商 API 進行部位對比，一旦發現平台部位與券商實體部位不符，立即觸發告警並鎖定該帳戶之交易功能。

## 5. 企業級風控體系與安全治理

風控中心作為平台的「交易塔台」，其設計原則為「安全性優於功能性」。在企業級運作中，系統必須具備抵禦極端行情與操作錯誤的能力。

### 三層風控防線

1. **Pre-trade (下單前)：** 檢查委託價是否在合理價格區間、單筆口數限制、保證金覆蓋率及 **Stale Quote (行情過時)** 偵測。
2. **In-trade (盤中監控)：** 監控即時權益數 (Equity) 波動、日損門檻 (Max Daily Loss)、券商連線心跳及異常重複下單阻擋。
3. **Post-trade (盤後對帳)：** 執行日終部位對齊、計算交易稅與滑價成本，並產生績效歸因報告。

### 安全治理架構

- **憑證管理：** 整合 HashiCorp Vault，所有券商 API Key 與機敏資訊均透過動態權限讀取，嚴禁明文存放。
- **安全標準：** 全系統對齊 **OWASP ASVS** 標準進行 Web 防禦，並建立具備 **Immutable WORM (一寫多讀) 儲存特性**的稽核日誌 (Audit Log)，確保所有人工修改策略參數或觸發 Kill Switch 的紀錄皆不可竄改。

## 6. 實作里程碑與開發團隊配置

轉型計劃採取分階段實施 (Phased Implementation)，以循序漸進地管理技術債並確保交易穩定。

### 七大里程碑實作路徑 (Phase 0–6)

- **Phase 0：法規與授權邊界定義** —— 確認法規合規邊界、權限矩陣、券商 API 可用性盤點。
- **Phase 1：基礎設施佈署** —— 完成 K8s 底座、分層資料庫、Vault 與監控系統 (Prometheus/Grafana) 建置。
- **Phase 2：資料平台與管線** —— 實現台指期歷史資料版本化、即時行情清洗與換月自動化管線。
- **Phase 3：策略開發與回測環境** —— 完成 Strategy SDK 與分散式 Backtest Farm，支援 Walk-forward 分析。
- **Phase 4：交易執行與風控核心** —— 實作 OMS、Risk Engine、Integer Optimization 調度模型與 Broker Gateway。
- **Phase 5：Web Command Center 與影子交易** —— 完成交易塔台 UI 頁面，並啟動 Shadow Trading (實盤行情、不送單) 驗證。
- **Phase 6：可靠性驗證與正式 Go-Live** —— 完成資安滲透測試、災難復原演練 (DR Test) 與小口數 (Small Live) 實盤驗收。

### 專業團隊配置評估

為確保此企業級平台穩定運行，建議配置以下職責角色：

- **Quant Lead：** 負責風險模型設計、連續月調整邏輯與回測重現性驗收。
- **Backend / Data Engineer：** 負責微服務開發、資料湖建置與 Event Sourcing 機制。
- **DevOps / SRE：** 負責 K8s 編排、CI/CD 與可觀測性系統 (Metrics/Logs/Traces) 維運。
- **Security Engineer：** 負責 Vault 密鑰管理、RBAC 權限稽核與 OWASP ASVS 安全合規。
- **UX Designer：** 針對 Command Center 與風險監控頁面進行高效能交易流程設計。
- **Legal / Compliance：** 負責確認交易行為符合期交所規範與內部合規審計。

**結語：** 本藍圖實踐了「策略只產訊號，平台負責執行、安全與資料治理」的設計哲學。透過雲原生架構與嚴謹的數學調度模型，將台指期量化交易從脆弱的腳本開發提升至具備韌性、安全、且可持續擴展的企業級平台。
